# JSON 超时问题 - 根本性修复完整说明

## 问题总结

**用户反馈**: "长时间输出json他说json不完整，那就是被截断了，因为llm还没有完整输入"

**表现症状**: 
```
错误: 超时：工具调用JSON不完整
```

## 根本原因分析

### 问题演变历程

**第一代修复** (v1.2.0):
- **问题**: 心跳消息中断JSON流
- **方案**: 流式JSON时禁用心跳
- **结果**: ✅ 防止心跳中断，但未解决时间限制问题

**第二代修复** (初版, 本PR前):
- **问题**: 2分钟超时太短
- **方案**: 改成5分钟
- **问题**: ❌ 治标不治本，仍是任意时间限制

**第三代修复** (本PR, 根本性):
- **问题**: 不应有任意时间限制，应基于LLM状态
- **方案**: 使用LLM生成状态检测，而不是轮询计数
- **结果**: ✅ 真正解决问题

### 为什么之前的方案有问题

```
原逻辑:
  if (轮询次数 > 600) {  // 5分钟后
    error('JSON不完整');
  }

问题:
- LLM正在第6分钟生成JSON → 仍会超时 ❌
- 2分钟内完成 → 浪费等待时间 ❌
- 不知道是LLM问题还是网络问题 ❌
```

### 根本原因所在

**LLM可能的状态**:
1. ✅ 正在生成 (UI显示进度/loading)
2. ✅ 已完成 (发送按钮可用)
3. ❌ 错误/卡住 (无响应且超时)

**之前的代码**只看时间，不看状态！

## 根本性修复方案

### 核心逻辑变化

```javascript
// 修复前: 时间驱动（错误）
if (pollCount > 600) {
  error('JSON不完整');
}

// 修复后: 状态驱动（正确）
var gen = isGenerating();
if (!gen) {
  // LLM已停止 + JSON不完整 = 真实错误
  error('工具调用JSON不完整（LLM已停止生成）');
} else if (pollCount > 1200) {
  // 防护: 即使LLM还在生成，也有10分钟限制
  error('流式传输超时（LLM生成超过10分钟）');
} else {
  // LLM仍在生成 = 继续等待
  log('继续等待...');
}
```

### 四种情况处理

| 情况 | LLM状态 | JSON完整 | 轮询数 | 处理 |
|------|---------|---------|--------|------|
| ✅ 成功 | 生成完 | 是 | 任意 | 返回结果 |
| ⏳ 等待 | 生成中 | 否 | <1200 | 继续轮询 |
| ❌ 错误 | 已停止 | 否 | 任意 | 报错 |
| ⏱️ 防护 | 生成中 | 否 | >1200 | 超时报错 |

## 实现细节

### 文件: `DeepSeekChatBridge.java`

**修改位置**: 行 612-639 (JSON工具调用处理)

**关键改动**:

1. **移除**: `if (pollCount > 600)` 的硬超时
2. **添加**: `var gen = isGenerating()` 状态检查
3. **分支1**: 当`!gen && !jsonComplete` → 报错
4. **分支2**: 当`gen && pollCount > 1200` → 防护超时
5. **分支3**: 当`gen && pollCount <= 1200` → 继续等待

### 调用的函数

**`isGenerating()`** (已存在, 行 400-431):
```javascript
function isGenerating() {
  // 检查UI中的生成指示符
  // - typing/loading/thinking 标志
  // - 发送按钮状态（暂停/停止图标）
  // - 最后消息的"生成中"文本
  return true/false;
}
```
